# Verdict — Case 08 (You Dirty Rat! — RAT Fingerprinting)

## 🔴 Verdict: TRUE POSITIVE

---

## MITRE ATT&CK Mapping

| Technique | ID | Description |
|---|---|---|
| System Information Discovery / Gather Victim Network Information | T1082 / T1590 | IP geolocation lookup (`ip-api.com`) to fingerprint the victim's environment |
| Masquerading | T1036 | Stale, hardcoded Chrome 73 (2019) User-Agent used to disguise a scripted request as ordinary browser traffic |
| Software Discovery / Virtualization & Sandbox Evasion (likely intent) | T1497-adjacent | Pattern consistent with malware checking for sandbox/hosting-provider ASN vs. genuine residential/corporate victim network before proceeding |

---

## Justification

1. **Deliberate environment fingerprinting** — the `GET /json/` request to `ip-api.com` returned the host's true public IP, precise city-level location, and ISP/ASN information (Google Fiber, Austin, TX). This kind of self-directed geolocation check has no legitimate business purpose on a corporate/lab workstation and is a well-documented technique used by RATs and loaders to distinguish real victims from sandboxes or research networks.
2. **Spoofed/stale User-Agent** — the request claimed to be Chrome 73 (a version current in early 2019) despite the capture being from mid-2024, a clear inconsistency indicating the request was generated by a script or malware sample with a hardcoded UA string, not an actual browser session.
3. **One-shot, no-session request pattern** (`Connection: close`) is consistent with a discrete recon step rather than organic user browsing.
4. **All higher-bandwidth traffic was independently ruled out** as legitimate (Maven Central, GitHub CDN, Windows/Microsoft telemetry), narrowing the suspicious activity down specifically to this fingerprinting call — it was not an artifact of noisy background traffic.

Consistent with the reasoning applied in prior cases (e.g., Case_06's spoofed User-Agent on C2 beacon traffic): a scripted, disguised HTTP request performing victim fingerprinting is itself sufficient indicator of compromise to close as TP, even though the full C2 command channel was not independently isolated in this session.

---

## What Would Change This Verdict to FP

- If the `ip-api.com` call could be attributed to an authorized internal tool or approved software with a documented legitimate need to geolocate the host (e.g., a licensing/compliance check) — no such context or supporting evidence was found.
- If the stale User-Agent could be explained by an old, but legitimate and approved, browser or embedded WebView still in use in the environment — considered unlikely given the one-shot `Connection: close` request pattern typical of scripted tools rather than an interactive browser session.

No such justification applies — verdict stands as TP.

---

## Recommended Response Actions

- Isolate host `DESKTOP-SKBR25F` (`172.16.1.66`) from the network
- Review endpoint/EDR logs to identify the process responsible for the `ip-api.com` request and confirm the RAT binary/loader
- Revisit the capture to complete C2 channel isolation (Duration-sorted TCP Conversations, looking for long-lived low-bandwidth connections not yet attributed to legitimate services)
- Block outbound access to `ip-api.com` and monitor for similar geolocation-lookup patterns from other hosts as a low-cost detection rule
- Check other hosts in the `wiresharkworkshop.online` domain for the same fingerprinting behavior, in case of broader compromise
